下面叙述第三个版本的基本思路

通过 OpenCV 给出来摄像头的接口，然后用 MediaPipe 做姿态识别，然后把姿态的信息传出给 call back 函数接口 `on_detected` 

传出的 `pose_info` 是由以下部分组成的一个 `dict` ：

1. `target_x: int` 中心横坐标
2. `target_y: int` 中心纵坐标
3. `height: int` 高度
4. `width: int` 宽度
5. `confidence: int` 置信度

具体实现如下：

- 中心位置：左髋(23) + 右髋(24) 的中点
- 高度：鼻尖(0) → 膝盖较低者(25/26)
- 宽度：左肩(11) ↔ 右肩(12)
- 置信度：双肩 + 双髋 visibility 平均值

这里采用的是把默认的 MediaPipe 的归一化坐标（左上原点，向右 x 轴，向下 y 轴）做了映射，现在的单位全部是像素点 px

### 目前存在的问题

#### 1. 多人检测策略不明确

当前选择"height 最大的人"，可能有以下几个问题：

1. 高度最大不代表它确实是人。有可能识别到置信度比较低（也即不是人），但是高度很高的类人
2. 两人之间切换时没有平滑过渡。有可能在两人之间快速抖动

#### 2. 缺少配置参数化

检测人数（num_poses）硬编码为 3
经测试，在同时检测 3 个人的时候，fps 会跌到 16 左右，但是如果调成只检测 1 个人，fps 会回升到 30 左右

摄像头 ID 硬编码为 0
置信度阈值无法外部调整

#### 3. 坐标转换问题

使用像素坐标，但如果更换摄像头分辨率，控制算法需要重新调参

#### 4. 模型选择

当前使用 MediaPipe 的 Lite 模型，精度 vs 速度的平衡是否合适需要实际测试

#### 5. 帧率的问题

当前的帧率不一定是摄像头的实际帧率。目前的帧率是在 call back 函数（在这里应该是 `test_pose_output.PoseTester.detection_callback` 函数）调用的时候，计算当时的帧率。但是由于那个 `if on_detected and pose_info:` ，只有在存在姿态信息也即有人像存在的时候才会调用回调函数，于是导致：如果当前帧镜头里没人，则当前帧不计数，于是帧率偏小

#### 6. 选择点的问题

高度：鼻尖不一定拍的到，膝盖也不一定拍的到
宽度：有可能转身会有问题，这个数据不一定可靠
置信度：由于双肩和髋部大概率拍的到，于是可能产生：高度没拍好，但是置信度很高的问题，所以置信度也有比较大的误差

   `deadband_*`：死区，在死区之内波动视为 0 ，==待调参==

   `pan_front & tilt_center`：云台的初态

   `lost_timeout_s`：判定失去跟踪的临界秒数，==待调参== 

3. `gimbal_controller` 云台控制的决策程序，这部分已经基本完善

传出的接口示例如下：

```python
{
'gimbal': 'gimbal', 
'pan': 3.222, 
'tilt': -2.001
}
```

其中，`pan` 和 `tilt` 表示目前云台 **需要转动的角度** 而不是要转动到的绝对角度

云台控制：
- 目前仅实现了 P 控制，I 和 D 两个控制需要后续补充，也可能好用的话就不补充了
- 基于动量实现了平滑
- 记录现在的绝对角度，为以后小车控制做准备，分别为 `_pan_abs` 和 `_tilt_abs` 
- 记录前一次的 `pan` 和 `tilt`，注意这里是相对值
- TODO: 后续可能还需要限制一次转头的最大角度

云台机械设计：
- 水平方向：`[-180, 180]` 也即 360 度范围，正前方为 0 度
- 竖直方向：`[0, 90]` 正前方为 0 度，正上方为 90 度
